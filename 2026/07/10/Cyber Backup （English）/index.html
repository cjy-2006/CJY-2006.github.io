Cyber Backup （English）

Project Introduction

People always want to hold onto something.

Some cannot let go of a relationship.

Some scour through chat histories, just wanting to read that one line again: “Did you get home safely?”

Others, when they finally turn back, find that the person who used to send 60-second voice messages is no longer there.

Yet voices fade.

Speaking habits are forgotten.

Even those most ordinary pauses, tones, and ramblings slowly sink into time.

Cyber Backup is a semi-automated personalized chat system.

It monitors input, records context, maintains memory, and calls on AI to automatically generate responses.

The system features clipboard monitoring, personality memory, anti-reflux mechanisms, chat rhythm segmentation, and automatic sending capabilities.

Unlike regular chatbots that spit out complete, cold blocks of text at once,

it mimics the pauses in real conversations.

The system gradually learns a person’s speaking style.

Short sentences, tones, pet phrases, and pause patterns all slowly accumulate in its memory.

You say one line.

It remembers one line.

It may not truly keep a person with you.

But years later,

when the chat box lights up again,

at the very least, that familiar way of speaking will emerge slowly, just like before.

Like an echo.

Like an unvanished old dream.

Frequently Asked Questions

1. Why does “self-talk (asking and answering oneself)” occur?

This happens because the responses generated by the bot are re-recognized as user input and rewritten into the inbox.

Reflux issues occurred frequently in the early stages of the project.

Subsequently, we resolved this through:

  • _bot_ tagging
  • Clipboard filtering
  • AHK pre-sending cleaning
  • Input anti-shake

and other methods.

2. Why are chat messages sent one by one?

The system actively splits long sentences.

Because real conversations do not involve sending large blocks of text at once, but rather include pauses, hesitations, and a sense of rhythm.

For example:

text

This effect is closer to real human conversation than outputting all at once.

3. Why are responses sometimes slow?

The system has an input buffering mechanism.

When the user sends short sentences continuously, the monitor waits for a few seconds until the input stabilizes before generating a unified response.

This is to avoid:

  • Interrupting the user
  • Context breakdown
  • Triggering AI once per sentence

4. Why is memory needed?

Memory is not intended to “remember everything long-term”.

Instead, it is used to:

  • Maintain consistent tone
  • Preserve personality
  • Record recent chat status
  • Simulate continuous dialogue

The system only reads the most recent part of the memory window by default.

5. Why not use official chat interfaces and instead write a custom monitor?

Because the project’s goal is not “chatting”.

The monitor, AHK, memory, and API together form a semi-automated personality system.

Initial version.

Only included:

  • API calls
